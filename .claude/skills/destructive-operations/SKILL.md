# destructive-operations スキル

ファイルやディレクトリの削除など、元に戻せない破壊的操作を安全に進めるためのスキルです。

## スキルの目的

- 削除操作の影響範囲を正確に分析し、ユーザーに伝える
- AI が破壊的操作を直接実行することを禁止する
- ユーザーが自分で実行するための手順を提示する

## 使用タイミング

- ユーザーがファイルやディレクトリの削除を依頼・相談したとき
- ユーザーがリポジトリ構成の整理を検討しているとき
- 破壊的な変更（ファイル移動・リネーム含む）の相談を受けたとき

---

## 🚨 絶対ルール

### NEVER（禁止）

- **AI がファイル・ディレクトリの削除コマンドを実行すること**
- `rm`, `Remove-Item`, `del` 等の削除コマンドをターミナルで実行すること
- `git rm` をターミナルで実行すること
- ユーザーの明示的な確認なしに影響分析を省略すること

### MUST（必須）

- 削除対象の影響分析を必ず行う
- 分析結果をユーザーに提示し、理解を得る
- 実行手順をコマンド付きで提示し、**ユーザー自身に実行してもらう**

---

## 影響分析チェックリスト

削除対象について、以下の観点で影響を分析する：

### 1. ローカル vs リモートの影響

| 確認項目 | 説明 |
|---------|------|
| ローカルファイルの削除がリモートサービスに影響するか | clasp pull で取得したファイルの削除は GAS プロジェクトに影響しない等 |
| Git 管理下か | `git status` で追跡状況を確認。Git で追跡されていれば履歴から復元可能 |
| 他のファイルから参照されているか | import、require、ファイルパス参照の有無 |

### 2. 環境依存の確認

| 確認項目 | 説明 |
|---------|------|
| 削除対象が prod / dev どちらの環境か | 環境の取り違えがないか |
| 環境固有の設定ファイルを含むか | `.clasp.json` 等、再取得が手間なファイル |

### 3. 復元可能性

| 確認項目 | 説明 |
|---------|------|
| Git にコミット済みか | コミット済みなら `git checkout` で復元可能 |
| リモートから再取得可能か | `clasp pull` 等で再度取得できるか |
| 完全に失われるデータはあるか | ローカルのみの未コミット変更 |

---

## 回答テンプレート

削除相談を受けた際は、以下の構成で回答する：

### ステップ 1: 影響分析の提示

```
## 影響分析

| 観点 | 結果 |
|------|------|
| リモートへの影響 | あり / **なし** |
| Git 追跡状況 | 追跡済み（復元可能） / 未追跡 |
| 他ファイルからの参照 | あり / **なし** |
| 復元方法 | git checkout / clasp pull / 復元不可 |
```

### ステップ 2: 実行手順の提示

ユーザーが自分で実行するためのコマンドを提示する。

```
## 実行手順（ご自身で実行してください）

1. 削除前の確認
   <確認コマンド>

2. 削除の実行
   <削除コマンド>

3. Git への反映（必要な場合）
   <git add / commit コマンド>
```

## 具体例

### ローカルディレクトリの削除（例: dev/ フォルダ）

**影響分析:**
- `dev/` フォルダは clasp pull で取得したローカルコピー
- 削除しても GAS プロジェクト（リモート）には影響なし
- Git にコミット済みなら履歴から復元可能
- 必要時に `clasp pull` で再取得も可能

**提示する実行手順:**

```powershell
# 1. 現在の状態を確認
git status

# 2. dev/ ディレクトリを削除
Remove-Item -Recurse -Force dev/

# 3. Git に反映
git add -A
git commit -m "chore: dev/ ディレクトリを削除"
```
