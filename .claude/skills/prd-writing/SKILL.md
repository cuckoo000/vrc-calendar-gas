# PRD作成スキル

このスキルは、高品質なプロダクト要求定義書(PRD)を作成するための詳細ガイドです。

## 前提条件

PRD作成を開始する前に、以下が完了している必要があります:

### アイデアの壁打ちが完了している

プロダクトのアイデアについて、ユーザー自身がClaude Codeとの対話を通じてブラッシュアップを完了している必要があります。

### docs/ideas/initial-requirements.md が作成されている

**重要**: 壁打ちの内容は、ユーザーが以下のファイルに保存する必要があります:

**ファイルパス**: `docs/ideas/initial-requirements.md`

このファイルには以下の内容が含まれている必要があります:
- プロダクトの基本的なアイデア
- 解決したい課題
- ターゲットユーザーの概要
- 実装したい主要機能
- MVPの範囲

PRD作成時には、このファイルの内容を参照して詳細化を行います。

## 既存ドキュメントの優先順位

**重要**: `docs/product-requirements.md` に既存のPRDがある場合、
以下の優先順位に従ってください:

1. **既存のPRD (`docs/product-requirements.md`)** - 最優先
   - プロジェクト固有の要件が記載されている
   - このスキルのガイドより優先する

2. **このスキルのガイド** - 参考資料
   - 汎用的なテンプレートと例
   - 既存PRDがない場合、または補足として使用

**新規作成時**: このスキルのテンプレートとガイドを参照
**更新時**: 既存PRDの構造と内容を維持しながら更新

## 出力先

作成したPRDは以下に保存してください:

```
docs/product-requirements.md
```

## テンプレートの参照

PRDを作成する際は、次のテンプレートを使用してください: ./template.md

## PRD作成プロセス

### 1. initial-requirements.md の確認

まず、ユーザーが作成した初期要求仕様を確認します:

```bash
Read('docs/ideas/initial-requirements.md')
```

### 2. PRDドラフトの生成

initial-requirements.mdの内容をもとに、テンプレートに従ってPRDを生成します。

### 3. PRDのレビューと改善

生成されたPRDを以下の観点でレビューします:

#### レビュー観点

1. プロダクトビジョンは明確か
2. ターゲットユーザーは具体的か
3. 成功指標は測定可能か
4. 機能要件は実装可能なレベルまで詳細化されているか
5. 非機能要件は網羅されているか

#### レビュー結果の評価基準

生成したPRDは以下の形式で評価します:

**✅ 強み**
- 明確なビジョンが測定可能で具体的に記述されている
- 機能仕様が実装レベルまで詳細化されている
- KPIが定量的な指標で定義されている

**⚠️ 改善が必要な点**

機能要件の曖昧さ:
- 問題: 具体的な実装仕様が不明確な箇所がある
- 推奨: 具体的なコマンド仕様やエラーハンドリングを明記

成功指標の測定方法:
- 問題: 測定方法が不明確
- 推奨: 測定方法とプライバシーへの配慮を明記

### 4. レビュー後の改善

レビューで指摘された問題を一つずつ確認し、具体化が必要な箇所を改善します:

1. 指摘された問題を一つずつ確認
2. 具体化が必要な箇所を改善
3. 改善後、再度レビューを実施
4. 問題がなくなるまで繰り返す

**注意点**:
- AIのレビューを鵜呑みにせず、最終的な判断は必ず人間が行う
- レビュー観点は明確に指定する
- 改善提案の妥当性を人間が検証する

## PRD作成の重要なポイント

### 1. 具体性と測定可能性

すべての要件は具体的で測定可能でなければなりません。

**悪い例**:
- システムは高速である必要がある
- ユーザーは使いやすいと感じる

**良い例**:
- コマンド実行時間: 100ms以内(平均的なPC環境で)
- 新規ユーザーが5分以内に基本操作を習得できる(ユーザビリティテストで測定)

### 2. ユーザー中心設計

すべての機能は、明確なユーザーの課題を解決するものでなければなりません。

**ユーザーストーリーのフォーマット**:
```
[ユーザー]として、[機能]したい。[目的]のために。
```

**例**:
```
ユーザーとして、習慣タスクに15分ブロックを記録したい。今日やった分を積み上げるために。
```

### 3. 優先順位の明確化

すべての機能に優先度を設定します:

- **P0(必須)**: MVP(Minimum Viable Product)に含める機能。これがないとプロダクトとして成立しない
- **P1(重要)**: 初期リリース後すぐに追加すべき機能
- **P2(できれば)**: 将来的に追加を検討する機能

### 4. ID管理とトレーサビリティ

機能・ユーザーストーリー・受け入れ条件間の対応を明確にするため、IDを付与します:

- **機能ID**: `F-01`, `F-02`, ... — コア機能一覧テーブルで管理
- **ストーリーID**: `US-01`, `US-02`, ... — ユーザーストーリーテーブルで管理

受け入れ条件は対応する機能ID(`F-01: [機能名]`)ごとにグループ化します。

## PRDの主要セクション詳細

### 1. プロダクト概要

#### 構成要素

1. **名称**: プロダクト名とサブタイトル
2. **プロダクトコンセプト**: 3つの主要コンセプト
3. **プロダクトビジョン**: 目指す世界観を3-5文で
4. **目的**: 具体的な目的のリスト

#### 例

```markdown
### 名称
**積み上げ(tsumi-age)** - 習慣タスク積み上げ記録アプリ

### プロダクトコンセプト
- 15分ブロック記録: 活動を15分単位で手軽に記録
- 積み上げビジュアル: 累計に応じたレベルアップで成長を実感
- シンプルな習慣管理: 余計な機能を省き、記録と振り返りに集中

### プロダクトビジョン
日々の習慣的な活動を15分単位のブロックとして記録し、
積み上げた努力を視覚的に実感できるアプリケーションを提供する。
レベルシステムによる成長の可視化で、継続のモチベーションを維持する。
```

**具体的な価値提案を含める**

悪い例:
```
便利なタスク管理ツールを作る
```

良い例:
```
日々の習慣を15分ブロックで記録し、積み上げを可視化するWebアプリ。

提供する価値:
- 記録の手軽さ(ボタン1つで15分ブロックを記録)
- 成長の可視化(レベルシステムによるモチベーション維持)
- 振り返りの容易さ(日別の記録履歴で実績を確認)
```

### 2. ターゲットユーザー

#### 必須要素

1. **ユーザー像**: 誰がどのような状況で使うか
2. **現在の課題**: 具体的な痛み点
3. **期待する解決策**: どうなりたいか

※ プロダクトの種類に応じて、詳細なペルソナ(年齢、職業、ワークフロー等)を追加してもよい。

#### 例

```markdown
### プライマリーユーザー
- 個人で習慣的な活動(英語学習、筋トレ等)を継続したい人
- 記録が面倒で習慣が途切れがちという課題を抱えている
- 手軽に記録でき、積み上げた成果を可視化してモチベーションを維持したい
```

### 2.5. ビジネス要件

WHY(なぜこのプロダクトが必要か)を明示するセクションです。

#### 必須要素

1. **課題**: ユーザーが抱えている具体的な問題
2. **解決策**: プロダクトが提供するソリューション

#### 例

```markdown
### 課題
習慣的な活動を継続したいが、記録が面倒で途切れやすく、
どれだけ積み上げたかが見えないためモチベーションが維持できない。

### 解決策
15分ブロック単位の簡単な記録と、レベルシステムによる
積み上げの可視化で、継続のモチベーションを維持する。
```

### 3. 成功指標(KPI)

#### SMART原則

- **S**pecific(具体的): 何を測定するか明確
- **M**easurable(測定可能): 数値で測定できる
- **A**chievable(達成可能): 現実的な目標
- **R**elevant(関連性): ビジネス目標と関連
- **T**ime-bound(期限): 達成期限を設定

#### フェーズ型の定義を推奨

成功指標はリリース時 → 短期 → 中期 のようにフェーズごとに定義すると、段階的な成長が計測しやすくなります。

#### 例

```markdown
### リリース時
- 基本機能が正常に動作する
- 1人以上のユーザーがタスクを作成・記録できる

### 1ヶ月後
- 週3日以上利用するアクティブユーザー: 5人
- ユーザーあたり平均タスク数: 3個以上

### 3ヶ月後
- 月間アクティブユーザー: 50人
- 継続率(1ヶ月以上利用): 40%以上
```

### 4. コア機能一覧・ユーザーストーリー・受け入れ条件

PRDでは以下の3つを**セクション分離**して記述します:

#### (1) コア機能一覧テーブル

全機能をID付きテーブルで一覧化し、全体像を把握しやすくします:

```markdown
| ID | 機能名 | 概要 | 優先度 |
|---|---|---|---|
| F-01 | ユーザー認証 | Google OAuth によるログイン/ログアウト | P0 |
| F-02 | タスクCRUD | 習慣タスクの作成・編集・削除 | P0 |
| F-03 | ブロック記録 | 15分ブロック単位での活動記録 | P0 |
```

#### (2) ユーザーストーリー

カテゴリごとにID付きテーブルで整理します:

```markdown
### 認証
| ID | ストーリー |
|---|---|
| US-01 | ユーザーとして、Googleアカウントでログインしたい。安全にアクセスするために。 |

### 習慣タスク
| ID | ストーリー |
|---|---|
| US-06 | ユーザーとして、習慣タスクを作成したい。継続したい活動を管理するために。 |
```

#### (3) 受け入れ条件

機能IDごとにチェックリスト形式で記述します:

```markdown
### F-01: ユーザー認証
- [ ] Googleの認証画面に遷移する
- [ ] 認証完了後、ダッシュボードにリダイレクトされる
- [ ] 未認証状態ではログインページにリダイレクトされる
```

### 5. 機能要件(技術仕様)

技術スタック、データモデル、ドメイン固有ロジック、セキュリティ要件を含めます。

**例**:
```markdown
### 技術スタック
| レイヤー | 技術 |
|---|---|
| フロントエンド | Next.js (App Router) + TypeScript |
| データベース | PostgreSQL (Supabase) |

### データモデル
**tasks テーブル**
| カラム | 型 | 説明 |
|---|---|---|
| id | UUID | 主キー |
| title | TEXT | タスク名(必須) |

### セキュリティ
- 全テーブルにRow Level Securityを適用
- サーバーサイドでの認証チェック必須
```

### 6. 非機能要件

テーブル形式で簡潔に記述します:

**例**:
```markdown
| 項目 | 要件 |
|---|---|
| レスポンス速度 | 初期表示: 2秒以内 |
| 対応ブラウザ | Chrome, Safari, Firefox, Edge(最新2バージョン) |
| レスポンシブ | 画面幅320px以上で正常表示 |
| アクセシビリティ | キーボード操作対応、適切なaria属性 |
```

## 品質基準とチェックポイント

PRDの品質を確保するため、以下のチェックポイントを確認してください:

### ビジョンと目標
- [ ] プロダクトビジョンが明確で測定可能か
- [ ] 提供する具体的な価値が定義されているか

### ターゲットユーザー
- [ ] ユーザー像と課題が明確か
- [ ] 期待する解決策が記述されているか

### ビジネス要件
- [ ] 課題と解決策が明確に対応しているか
- [ ] WHY(なぜこのプロダクトが必要か)が伝わるか

### 成功指標
- [ ] フェーズごとのKPIが定義されているか
- [ ] 測定可能な数値目標が設定されているか

### 機能要件
- [ ] コア機能がID付きテーブルで一覧化されているか
- [ ] ユーザーストーリーがID付きで整理されているか
- [ ] 受け入れ条件が機能IDごとにチェックリスト形式か
- [ ] 技術スタックとデータモデルが定義されているか

### 非機能要件
- [ ] パフォーマンス・対応環境等がテーブル形式で定義されているか
- [ ] セキュリティ要件が明確か

## まとめ

PRD作成の成功のポイント:

1. **initial-requirements.md をベースに作成**: ユーザーが作成した壁打ち内容を参照
2. **構造化とID管理**: 機能一覧(F-xx)・ストーリー(US-xx)・受け入れ条件をセクション分離し、IDで紐付け
3. **ビジネス要件の明示**: 課題 → 解決策でWHYを明確に
4. **フェーズ型KPI**: リリース時 → 短期 → 中期で段階的に目標設定
5. **具体性と測定可能性**: すべての要件を明確に
6. **ユーザー中心**: ユーザーの課題を解決する機能のみ
7. **レビューと改善**: 自己レビューと人間による最終判断
